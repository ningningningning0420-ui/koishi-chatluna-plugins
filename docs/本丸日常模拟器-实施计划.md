# 本丸日常模拟器（chatluna-life-sim）实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: 用 superpowers:subagent-driven-development（推荐）或 superpowers:executing-plans 按任务逐个实现。步骤用 `- [ ]` 复选框跟踪。
> **配套设计文档**：`docs/本丸日常模拟器-设计文档.md`（定稿）。本计划的每个任务都以该文档为准，条目后标注对应章节 §。

**Goal：** 给每个刀剑男士 agent 进程补上「② 日常生成层」，让 bot 闲时自己过日子、长出持续一致的不在场生活；再补「① 协作层」让多 agent 经毛线球互动。

**Architecture：** 系统**已是多 agent**（4 个独立 koishi 进程/DB/QQ）。本插件 `koishi-plugin-chatluna-life-sim` 装进**每个 app 进程内**跑「观察→roll→长短期记忆→规划→反思→主动」自循环（P1/P2，纯进程内，用该 app 自己的 sqlite DB）；协作层引入**毛线球**协调/路由进程 + 各 bot 自产对话（P3，跨进程共享存储）。记忆两层直取、不掺 livingmemory 向量；时间=现实时间，只处理并发；决策交人设、机械只兜极端。

**Tech Stack：** koishi（cordis）、minato ORM（`ctx.database.extend`）、`@cordisjs/timer`（`ctx.setTimeout`）、ChatLuna（`ctx.chatluna.createChatModel`）、LangChain message 类、chatluna-character 事件、worldbook 纯函数、JSON-schema 约束输出。

**语言/测试约定（仓库现状，必须遵守）：**
- **纯 JavaScript（CommonJS）**，`'use strict'` + `require` + `module.exports`；**不用 TypeScript、无 tsconfig、无构建步骤**。`package.json` 的 `main: index.js`。
- **文件划分照 relay/memory-curator 现状**：`index.js` = koishi 插件胶水（注册配置/建表/装配/生命周期）；纯逻辑拆到 `lib.js` 或按域拆多个 `*.js`；模型/DB 封装各一文件。
- **测试 = 每包一个离线 `test.js`**：手写 `test(name, fn)` + `node:assert`，`node test.js` 跑（无框架，"matches the project's existing offline-self-test habit"）。纯逻辑函数 red→green 都进 `test.js`。
- **下述任务里凡写 `src/xxx.ts` 一律实现为仓库风格 `xxx.js`（CommonJS）；凡写 `tests/xxx.spec.ts` 一律为追加到本包 `test.js` 的离线自测。** 路径中的 `src/`/`tests/` 目录按需扁平化或建轻量子目录，跟随本包既有布局。

## Global Constraints

- **不改运行代码前，本计划只是文档**；执行时新建插件包，不动现役 app 配置。
- **每个 agent 用自己的 DB**：所有表经 `ctx.database.extend` 建在本 app 的 `data/koishi.db`；P3 的共享存储另置于 `sharedStorePath`（独立文件）。§3.0
- **时间 = 现实时间**：无 clockScale；不做 gap 补档；关机段=「不在」。§5.11
- **记忆不写 livingmemory 向量**：sim-memory 自成两层、直取为主；只借鉴其 dream 整理思路。§4.6
- **决策交人设 + 可开关硬熔断兜底**：事件/外联/追问/互聊收尾都由 bot 自判；机械帽（quietHours/dailyCap/botChatMaxTurns）仅熔断。§5.8/5.13/3.3
- **模型调用照搬 photo 插件** `model.js` 的 `ctx.chatluna.createChatModel(platform,name).invoke(lcMessages,{signal})`；`rollModel` 默认本地 ollama，`consolidateModel` 用主模型。§2
- **纯逻辑单元必须 IO-free + 离线自测**（进 `test.js`）：`matchContext`/weather markov/`nextWake`/连续性 clamp/gate 判定/短→长聚类打分/dedup。§4.9/5.11/5.12
- **命名/署名**：包 `koishi-plugin-chatluna-life-sim`，作者妖祀，跟随仓库现有发布元数据风格。
- **提交**：TDD、频繁提交；本仓 post-commit 自动推送，先在 `feat/life-sim` 分支上做。

# Phase 1 — ② 日常生成层落在单 agent（execution-ready）

> 装进一个 app（先髭切）进程内。验收：开一两天后 `life_sim_event` 有贴合日程、随天气/时段变化的连续事件流；长期库有夜间沉淀条目；下次聊天能自然、**一致**地提起这两天做的事（追问对得上）；对话中断不显冷；主动消息符合人设节奏。§9

## 文件结构（P1，纯 JS/CommonJS，照仓库现状）

**路径映射规则**（下面各任务的 `.ts` 路径按此翻译）：`src/a/b.ts → a-b.js`（包根扁平，斜杠换连字符）；`src/x.ts → x.js`；所有 `tests/*.spec.ts` → 统一进包根 `test.js`。

## 任务分解（P1，按依赖排序）

### Task 1: 插件骨架 + 配置 schema + 建表 + 模型封装
**Files:** Create `package.json`, `src/index.ts`, `src/config.ts`, `src/db.ts`, `src/model.ts`; Test `tests/db.spec.ts`, `tests/model.spec.ts`
**Interfaces:**
- Produces: `Config`（§8 全部字段，带默认值）；`declare module 'koishi'` 建表 `life_sim_task/state/event/world/plan/assignment/thought/ltm/relationship/routine`（schema 见设计文档各节）；`getModel(ctx, 'platform/name')→ChatModel`、`invoke(model, messages, {signal})→string`（照搬 photo `model.js`）。
**Deliverable:** 插件能加载、建表成功、能用配置里的 `rollModel` 跑通一次 `invoke`（冒烟）。
**Tests:** `getModel` 解析 `platform/name`；`invoke` 把 `{role,content}[]` 转 LangChain 消息并取回文本（可 mock model）。
**Steps:** 写 config schema 默认值测试 → 写 db 建表断言 → 写 model 封装测试(mock) → 实现 → 冒烟跑通 → commit。

### Task 2: 持久化调度 Scheduler + 单活动锁 ConcurrencyGuard
**Files:** Create `src/scheduler.ts`; Test `tests/scheduler.spec.ts`
**Interfaces:**
- Consumes: `life_sim_task` 表、`ctx.timer`。
- Produces: `scheduleTask(presetId, fireAt, type, payload?)`、`onReady()`（从"现在"续排，过期丢弃/单次收尾，不补档 §5.11）、`dispose()`（清内存 timer、DB 保留）；`ConcurrencyGuard.acquire(presetId, kind)/release/isBusy`（kind: roll|peerchat|withuser；同 preset 单活动 §5.11）。
**Deliverable:** 任务落库、重启从现在续排、单活动锁互斥。
**Tests（纯逻辑）:** 过期任务不逐条补（只续下一次）；`acquire` 同 preset 二次占用返回 false；`isBusy` 正确。
**Notes:** `ctx.setTimeout` 内存态，disposer 存 Map；`ready` 扫 `{status:'pending'}`。§5.9

### Task 3: PresenceState 交接（WITH_USER/LINGERING/LIVING）
**Files:** Create `src/presence.ts`; Test `tests/presence.spec.ts`
**Interfaces:**
- Consumes: `chatluna_character/message_collect`（或 chat 事件）、Scheduler、ConcurrencyGuard。
- Produces: `state(presetId)→'WITH_USER'|'LINGERING'|'LIVING'`；`onUserMessage(session)`（→WITH_USER，抢占，打断当前活动标 interrupted）；`onQuiet()`（排 `lingerWindowMin` 评估 wake → LINGERING）；`recordActivity`。
**Deliverable:** 用户消息抢占、静默进 LINGERING、评估 wake 触发。§5.13
**Tests（纯逻辑）:** 状态机转移表（用户消息任何态→WITH_USER；LINGERING 超时→LIVING）；LINGERING 内不发 roll 任务。
**Notes:** 回复本身走既有 chatluna-character 管线，本单元只让路 + 管状态 + 排评估 wake。

### Task 4: WorldContext（时钟=现实时间 + 天气 markov + 季节）
**Files:** Create `src/world/context.ts`; Test `tests/world-context.spec.ts`
**Interfaces:**
- Produces: `getWorld(presetId)→{clock,timeOfDay,season,weather,locations,externalLocations}`；`tickWeather(prev, table)→next`（markov 相邻转移，纯函数）；`timeOfDayOf(clock, timezone)`、`seasonOf(date)`。
**Deliverable:** 世界状态可读；天气按步长演化、只相邻转移。§5.4
**Tests（纯逻辑）:** markov 只产相邻天气（晴不直接→暴雪）；timeOfDay/season 从真实时钟+timezone 派生正确。

### Task 5: EventRegistry（动态事件类型 + 连续性约束）
**Files:** Create `src/world/registry.ts`, `src/world/continuity.ts`; Test `tests/registry.spec.ts`, `tests/continuity.spec.ts`
**Interfaces:**
- Produces: `loadRegistry(path)→Entry[]`（热重载）；`available(world, lifeState)→WeightedType[]`（`matchContext` 过滤 + 权重）；`sampleType(weighted, rngSeed)`；`continuityClamp(nextState, world)→{ok, clamped, reason}`（时钟单调/地点合法/移动花时间/§5.4c）。
**Deliverable:** 按 WorldContext 筛出可发生类型集、加权抽样；非法状态被夹回。
**Tests（纯逻辑）:** `matchContext` 各条件组合（时段/天气/季节/requiresOtherPresent）；连续性 clamp 拦瞬移/时钟倒流/未授权外部地点。

# Phase 2 — 打磨与深化（task-outline，落地时细化）

> 依赖 P1 实机。每条一个可独立测试的交付物。

### P2-T1: 反思 Reflector（§5.6）
**Files:** `src/reflect.ts`。**产出**：`reflect(presetId)`——累积重要度过阈或夜间触发 → 取最近 N 条 → 模型抽"高层问题"→ 检索相关 → 合成洞见写回长期（可成树）。**测试**：触发阈值判定（纯逻辑）；洞见落 `life_sim_ltm(kind=insight)`。

### P2-T2: 长期记忆压缩/遗忘 + 可选 embedding（§4.5）
**Files:** `src/memory/forget.ts`, 扩展 `long.ts`。**产出**：dream 式内部再聚合（merge/archive）；`importance×refCount` 遗忘；`ltmEmbeddings:true` 时给长期建 embedding + `relevance` 联想深取（附加，非骨干）。**测试**：遗忘保留高 refCount；WeeklyPruner 清阈下残留。
