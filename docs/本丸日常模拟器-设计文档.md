# 本丸日常模拟器（chatluna-life-sim）设计文档 · 定稿

> 状态：设计定稿（不改运行代码）
> 日期：2026-06-30
> 作者：妖祀 / Claude
> 一句话：**本丸已经是多 agent**（每个刀剑男士 = 一个独立 koishi 进程/DB/QQ 账号）。本设计**不新建架构**，只在既有多 agent 之上补两层——**② 日常生成层**（每个 agent 自己"过日子"：观察→roll→记忆→规划→反思的后台循环）和 **① 协作层**（让这些独立进程共享世界、能互动、一起经历事件）。斯坦福小镇 Generative Agents 只作①②的实现参考。目标：长出一段**持续、一致、可被引用**的不在场生活，「不聊天，世界也在长」。

## 0. TL;DR

- **定位（重要）**：系统**已经是多 agent**（4 个独立 koishi 进程：髭切/膝丸/(空)/鹤丸国永，各自独立 DB + QQ，仅靠 relay 互相寻址）。所以**不是"搭 multi-agent"**，而是补两层：**② 日常生成层（每 agent 进程内，P1/P2）** + **① 协作层（跨进程，P3）**。斯坦福小镇只当①②的实现参考，不替换已有架构（§3）。
- **记忆分两层，全程直取，不靠向量彩票**：短期（工作集，按 recency 直接注入）+ 长期（按 key 直取，relevance 向量仅作可选联想）。斯坦福的 recency/importance/relevance 三信号在这里**各司其职**（§4）。
- **记忆分两层，全程直取，不靠向量彩票**：短期（工作集，按 recency 直接注入）+ 长期（按 key 直取，relevance 向量仅作可选联想）。斯坦福的 recency/importance/relevance 三信号在这里**各司其职**（§4）。
- **与 livingmemory 分域协同**：livingmemory 管「和用户对话」的记忆（不动）；sim-memory 管「不在场生活」的记忆（自成两层、自己注入、不写它向量）。沉淀逻辑复用 livingmemory 的 dream 思路，各库各跑（§4.6）。
- **它比"模型幻觉"多的唯一硬价值**：**跨时间一致 + 闭合因果**。幻觉每次编得都不一样、无法对质；模拟器记的是同一段、你能引用的历史（"你不是说在读那本书？"对得上）。要实现它，恰恰**不能靠向量检索低重要度日常**，而要靠直取式短期工作集 + 按 key 直取的长期档案（§1、§4）。
- **行动受约束**：JSON schema 框结构 + life-state 锚 + 每次重注人设 canon（查 worldbook）+ 计划当主干 → 让「半自由叙事」既生动又不崩人设。世界有**时空连续性**（时钟单调/移动花时间/天气相邻转移）；出本丸走**许可制 + 外部世界书**（§5）。
- **主动外联**：无 importance 门槛，发不发/何时发/没回是否再发全交人设判断；只留两道**可开关**硬熔断（安静时段 + 每日上限）（§5）。
- **① 协作层（P3）**：新增**毛线球**协调/路由进程（世界外，**不生成内容**）；**对话由各 bot 用自己完整语境真实生成**（peer 路径，不跑用户窗口），**收尾由 bot 自己决定**，毛线球只撮合/路由/持单一权威 transcript/熔断兜底/分发；关系档案各 bot 本地存自己视角。这样关系不脱离 bot 本体、对话更自由（§3.3）。
- **MVP**：P1 = **② 日常生成层**单 agent 落地；P2 反思/长期压缩/周报/突发·节日；P3 = **① 协作层**（毛线球 + 各 bot 自产对话）（§9）。

## 1. 价值判断

### 1.1 它真正解决什么，以及"为什么不是模型幻觉就够了"

当前 bot 的连续生活感完全依赖被动对话——不聊就不长。但**关键的反问是：你问"最近在干嘛"，模型张口就能编，凭什么要建这套东西？**

诚实的答案：**只要你从不回头核对，幻觉确实够用。** 模拟器唯一、且幻觉根本做不到的价值是——

- **跨时间一致性**：幻觉每次给的答案都不同（今天说在读兵法书，明天问"读完没"就露馅）。模拟器记的是**同一段、持续、你能拿来对质**的历史。
- **闭合因果**：「约了明天对练」→ 明天真发生 → 后天你提起对得上。幻觉没有持久状态，每次提及都是孤立的，闭不了环。
- **跨 agent 一致**：髭切和膝丸对同一段共同经历记得一致。两个独立模型上下文靠幻觉无法协调。

### 1.2 这决定了一条硬设计约束：别靠向量捞低价值日常

日常生活记忆**重要度天然低、语义又不一定贴近当前话题**——正是 embedding + importance 排序检索最容易埋掉、抓不到的那类。所以"归档进向量库等它自然浮现"是错的路。**正确路径是直取**：最近的按 recency 直接注入（永远在场），关系/线索按 key 直取。低重要度不再是问题，因为我们不拿重要度来 gate 日常注入（§4）。

### 1.3 有没有更简单的替代

| 替代 | 成本 | 一致性 | 结论 |
|---|---|---|---|
| 纯模型幻觉（问就编） | 0 | **无**（每次不一样） | 你若从不对质，够用；要连续/可引用则不行 |
| 纯随机事件池（不调模型抽签） | 0 | 有一致但**不长、重样** | 当冷启动/降级兜底 |
| **本设计（agent 循环 + 直取式长短期记忆）** | 低（便宜模型/本地免费） | **有，且可引用** | 推荐 |

→ 纯随机池保留为可切换的**兜底挡位**；默认走 agent 循环。

## 3. 架构：既有多 agent 之上补①协作层 + ②日常生成层

### 3.0 现状：已经是多 agent（不是要新建的东西）

本丸**当前就是多 agent 架构**——而且是"最硬"的那种：

| app | 角色 | 进程 | DB | 账号 |
|---|---|---|---|---|
| koishi-app | 髭切 | 独立 koishi 进程 | 自己的 `data/koishi.db` | 独立 QQ |
| koishi-app2 | 膝丸 | 独立进程 | 独立 DB | 独立 QQ |
| koishi-app3 | (空/未激活) | 独立进程 | 独立 DB | 独立 QQ |
| koishi-app4 | 鹤丸国永 | 独立进程 | 独立 DB | 独立 QQ |

- 每个刀剑男士 = **一个独立 koishi 进程 + 独立 sqlite DB + 独立 QQ 账号**，各带 chatluna-character/livingmemory/worldbook/scene-rules/relay。
- **agent 间当前唯一通路 = relay**（`[[relay:alias|…]]` 按 alias→QQ 发真实消息），且只在人为/上下文触发时发生。
- **缺的只有两块**：agent 之间不协作、不共享世界、不自主互动（缺①）；没有后台日常生成（缺②）。

→ **本设计不"搭 multi-agent"，只补两层。** 因为是**跨进程 + 各自独立 DB**，两层的实现方式截然不同：

- **② 日常生成层（每 agent 进程内，容易，P1/P2）**：作为插件装进每个 app，用该 agent **自己的 DB** 跑「观察→roll→长短期记忆→规划→反思→主动」。纯进程内，零跨进程依赖。
- **① 协作层（跨进程，是真正新的基础设施，P3）**：让独立进程共享一个本丸世界、能互动、一起经历事件。因跨进程跨 DB，必须引入**共享存储 + 协调者**（§3.3）。

**斯坦福小镇只作①②的实现参考**：其「观察/记忆/反思/规划/行动」循环 → 参考给②；其「相遇→对话→写回记忆」→ 参考给①（落到跨进程现实），**不拿来替换已有架构**。

### 3.1 ② 日常生成层：每个 agent 进程内的循环

```
   ┌──────── 一个 app（一个刀剑男士 agent 进程）内，life-sim 插件 ────────┐
   │ 闲置>idle(30min)/持久化调度触发                                       │
   │ ① 观察 Observe: roll事件 / WorldContext / 收到用户消息 /（P3:别的agent） │
   │        ▼                                                              │
   │ ② 记忆流 Memory: 短期(工作集,recency直取) ⇄ 长期(key直取+可选联想)      │
   │        ▼                                                              │
   │ ③ 反思 Reflect: 累积重要度/夜间 → 关系·自我认知 → 写回长期              │
   │        ▼                                                              │
   │ ④ 规划 Plan: 自著常规日程 + 被安排 → 每日计划(主干) → 打断重规划         │
   │        ▼                                                              │
   │ ⑤ 行动 Act: 便宜模型 roll 行动/事件(计划主干,list-then-roll)            │
   │        │ 输出: event + plan_adherence + want_to_share + next_state + next_delay │
   │        ├─→ 本地 sim-memory（该 agent 自己的 DB）                        │
   │        ├─→ 主聊注入: {recent_life}{life_state}{today_plan}{pending_thoughts} + 既有{living_memory} │
   │        └─→ (want_to_share=now) ProactiveBridge → relay 发用户            │
   │ [持久化 Scheduler]: 本地 DB 落库 + 启动扫描，重启不失忆                  │
   │ [WorldContext]: ②阶段为本 agent 私有；①阶段提升为跨进程共享（§3.3）      │
   └──────────────────────────────────────────────────────────────────────┘
```
**全部在一个 agent 进程内、用它自己的 DB**——所以能先落地（P1）。斯坦福循环是这一层的参考。

### 3.3 ① 协作层：毛线球（协调/路由，不生成）+ 各 bot 自己产出对话（P3）

因为 agent 是**独立进程 + 独立 DB**，协作需一个**专用协调进程「毛线球」**（世界外基础设施，第 5 个进程，headless，无 QQ/无人设）。4 个 app 同机 → 共享存储用**共享目录/共享 sqlite**（独立于各 app `koishi.db`）。斯坦福「相遇→轮替对话→某一方自己决定结束→写回记忆」是这层的直接参考。

> **核心原则：毛线球不生成一个字。** 对话内容由**每个 bot 用自己的完整语境真实生成**（自己的人设 + 自己的记忆 + 当前心情/近况 + 自己的模型），跑在一条**独立于用户窗口的 peer 路径**上（不进 chatluna-character buffer / livingmemory extraction，不污染对话域）。**关系与记忆是 bot 自己的产出，不脱离本体。** 毛线球只做协调/路由/持权威 transcript/分发——不当编剧、也不当裁判（收尾归 bot，见下）。

> **毛线球 ≠ 近侍**：**近侍**是**世界内角色**（某把刀，管日程治理，§5.3e）产出"安排意图"；**毛线球**是**世界外进程**，机械协调。名字取意：agent 之间的关系与未了「线」都由这颗球串起路由。

```
              ┌──────── 毛线球（协调/路由进程，世界外，headless，不生成内容）────────┐
              │ 仲裁: 接收 agent wake-up 的"去找X"意图 → 查对方状态 → 社交结果        │
              │ 路由: 轮到谁→通知谁生成；收其 turn→追加进权威 transcript→转发对方     │
              │ 熔断: 只在跑不停/死循环时按硬安全帽强制关（兜底，非正常收尾）           │
              │ 分发: 结束后把权威 transcript 投各 agent 收件箱                        │
              │ 托管 [本丸共享存储]（同机目录/sqlite，独立于 koishi.db）              │
              │   ├─ 共享 WorldContext（权威 时钟/天气/季节，共读一份）              │
              │   ├─ 在场登记（谁在线/谁忙，供仲裁）                                 │
              │   ├─ 进行中对话的权威 transcript                                     │
              │   └─ 交互队列 / 每 agent 收件箱                                      │
              └──▲──────────────▲────────────────────────┬──────────────────────────┘
       "去找X"意图 / 我这句turn│   近侍(角色)安排意图│              │轮询摄取回本地 sim-memory
        [各 app: 自己生成turn]─┘   [近侍所在 app]────┘        [各 app: 自己抽记忆/更新关系]─┘
```

**唯一心跳 = agent 的 wake-up 自循环**：互动"发起"不是毛线球另设的时钟，而是某次 wake-up 里 roll/计划产出的行动（"去找膝丸"）。近侍的安排也只是落成 Assignment 进计划、到点 wake-up 浮现——同一条路，不另起并行触发。

**一场对话的流程**：
1. **发起**：A 的 wake-up 产"去找 B"意图 → 毛线球。
2. **社交仲裁**：毛线球查 B 状态 → 聊 / 加入 / 打招呼即走 / 改约（§5.11 点 3）。若聊：
3. **开场**：毛线球告诉 A、B"你俩开始对话，对方是谁"；各自读**自己本地的关系档案**（自己视角）。
4. **轮流各自生成**：轮到谁，**谁在自己进程用自己完整语境生成一句**（peer 路径）→ 发回毛线球 → 追加进**唯一权威 transcript** → 转给对方"对方说了 X，该你"。
5. **收尾 = bot 自己的行为**：每次生成时，bot 可决定**收尾**（自然告别 + 结构化 `end` 信号），由人设/情境定（说完了 / 腻了 / 要忙别的 / 见对方要走）。**一方收尾则对话自然结束**——这是 Smallville 的"某一方决定结束"。毛线球**不裁**正常收尾，只在**跑不停/echo 死循环**时按硬安全帽 `botChatMaxTurns`/成本上限**强制关**当熔断兜底。
6. **收尾后**：毛线球把权威 transcript 投各 agent 收件箱；**每个 bot 从同一 transcript 抽自己的记忆**（自己的产出、自己的视角，条数按内容定）+ 更新**自己本地的关系档案**。事实由单一 transcript 对齐，视角/心情按各自人设分化。

**为什么这版对（相对旧·集中式）**：旧版毛线球拿压缩卡替双方生成 → 对话和关系是**毛线球的产出、脱离 bot 本体**；新版**生成交回各 bot**（带真实人设/记忆/心情，更自由），毛线球只协调/路由/持权威 transcript。防 echo 反而更好（各自真人设生成，不是一个模型 juggle 两张卡）；一致性仍由单一权威 transcript 保证；收尾更自然（bot 自己决定，非机械 cut）。

**relay 真互动（围观模式）**：`visibleVia: relay` 时 agent 之间真发 QQ 消息/群聊、各跑自己管线，可见可回放；贵，仅"想围观"时用。

**不污染对话域**：peer 路径独立于用户窗口——bot 生成 turn / 抽记忆都走 sim-memory 域，不进 chatluna-character buffer / livingmemory extraction。

**为什么不走"私聊/群聊窗口"执行**（次优）：走用户窗口 = 每轮跑完整 chatluna-character 管线（worldbook/livingmemory extraction/buffer）为背景闲聊空转，且把 bot↔bot 消息抽成"对话记忆"污染对话域。**保留对的部分**（各 bot 用自己人设 + 便宜模型 `peerModel` 真实生成），**换掉执行路径**（用户窗口 → 独立 peer 路径 + 毛线球路由；relay 仅当围观传输）。

### 3.4 单元清单（隔离设计，可独立测试）

## 4. 记忆系统：长短期两层（核心）

### 4.1 短期记忆 STM（工作集）

- **存什么**：最近的观察/事件、当天日程进度、刚发生的交互、当前 mood/state。
- **表示**：全量逐条事件（`life_sim_event`）+ life-state 对象（§5.2）+ today-plan + 逐句 transcript（若有交互）。
- **生命周期**：滑动窗口（最近 `stmDays` 天 / `stmMax` 条）；旧项在夜间沉淀进长期后滚出丢弃。
- **怎么取**：按 **recency 直取**，**永远在场、直接注入**（`{recent_life}`/`{life_state}`/`{today_plan}`）——不评分、不向量。这覆盖"日常连续性"，正是向量最容易漏的部分。

`life_sim_event`（当天/近期缓存）：
```
id, presetId, day, ts, title, narrative, event_type, location, participants(json),
mood, duration_min, importance(0-1), threads(json), plan_adherence, type, consolidated(bool), sourceModel
```

### 4.2 长期记忆 LTM

- **存什么**：沉淀后的重要事件、**关系档案（自己视角，每个对手一条）**、未了线/习惯、反思洞见、自我认知。
- **表示**：蒸馏条目（带 importance/keywords/entity-key）+ 关系档案 + 反思节点。
- **生命周期**：由短→长沉淀生成（§4.3）；长期内部再 **dream 式压缩**（keep/update/merge/archive）；按 importance + 引用计数遗忘（§4.5）。
- **怎么取**：主按 **key 直取**——`relationship(other)`（本地自己视角）/ `thread(id)` / `entity(name)` / `date(周年)`；**relevance 向量仅作可选联想深取**（给长期库建 embedding 当附加索引，非骨干）。

`life_sim_ltm`（长期条目）：
```
id, presetId, kind(event|insight|habit), content, summary, keywords(json),
entities(json), importance, refCount, createdAt, lastAccessedAt, embedding?(可选)
```
`life_sim_relationship`（关系档案，**存各 bot 自己本地、是自己视角**，直取）：
```
presetId(我), otherKey('膝丸'), summary(我对这段关系的滚动摘要),
openThreads(json), tone(我眼里的关系基调), lastChatId, updatedAt
```
→ 髭切存"我眼里的膝丸"、膝丸存"我眼里的髭切"，两份各自本地、都从同一权威 transcript 派生（事实对齐、视角分化）；不放毛线球共享，符合"关系由 bot 自己产出"。

### 4.3 短→长沉淀（consolidation，复用 livingmemory dream 思路）

- **时机**：夜间「睡觉」（`sleepHour`）+ 反思触发（累积重要度过阈，§5.4）。
- **过程**（作用于 sim 自己的库，算法family 同 livingmemory dream）：
  1. 取当天短期事件流 + 交互 transcript（含互聊后各自抽的记忆）；
  2. 便宜/主模型**聚类 + 决策 keep/update/merge/archive**（按 keywords/entity/时间聚类，importance 加权）；
  3. 生成/更新**长期条目**、**关系档案**（本地自己视角，滚动摘要 + 未了线）、**反思洞见**；
  4. 短期窗口旧项滚出丢弃（已蒸馏进长期）。
- **风格对齐**：长期 `content` 用第一人称、≤100 字、keywords≤12、不堆日期——和 livingmemory 条目同构，便于将来互操作。

### 4.4 检索：斯坦福三信号各司其职（"为什么不靠向量彩票"的正解）

斯坦福把 `recency+importance+relevance` 加权用于**每一次**检索。我们**拆开分工**：

- **recency** → 决定**短期工作集**：最近的永远直接注入，无需检索。→ 覆盖日常连续性。
- **importance** → 决定**谁沉淀进长期、谁被遗忘**（生命周期），**不**拿来 gate 日常注入。
- **relevance（向量）** → 只在需要"联想很久以前的旧事"时，对**长期库**做**可选**深取；不是日常主干。

→ 低重要度日常永远靠 recency 直取覆盖，绝不进彩票；向量只服务稀有的深度联想。这条是整套记忆设计的定盘星。

### 4.5 遗忘 / 压缩（sim 自己的"dream"，直取式）

- 短期：滑动窗口，旧的沉淀后滚出。
- 长期条目：dream 式聚合（merge/archive）；遗忘按 `importance × 引用计数`（被反复触及的 thread/关系不删，参考 Letta reference-count 优先）。
- 关系档案：定期 LLM 重新滚动摘要 + 未了线（解决即关）。
- transcript：近的留逐句，旧的压缩/归档（关系档案已留蒸馏版）。
- 每周 `WeeklyPruner`：清 `importance < pruneThreshold` 且已沉淀的短期残留。

### 4.6 与 livingmemory 的边界与协同

| | livingmemory | sim-memory |
|---|---|---|
| 域 | 和**用户对话**的记忆 | bot **不在场生活**的记忆 |
| 检索 | vector + importance（**不动**） | 直取为主 + 可选联想 |
| 注入变量 | `{living_memory}` | `{recent_life}/{life_state}/{today_plan}/{relationship}/{pending_thoughts}` |
| 整理 | 自带 dream | 复用同思路，各库各跑 |

**四个协同点**：
1. **并排注入互补**：主聊 prompt 里 livingmemory 管"聊过啥"、sim-memory 管"私下经历啥"，互不污染。
2. **整理逻辑一致**：sim 沉淀复用 livingmemory 的 dream 算法family（keep/update/merge/archive + 重要度聚类），保证两套系统"整理思路统一"，但**各管各的库**。
3. **自然回流**：某生活事件**真被拿到对话里聊** → 那次对话经正常 chat pipeline **自然**成为 livingmemory 条目。重要生活事件因此以自然方式进入"关系记忆"，**sim 不硬写 livingmemory**。
4. **可选联想**：将来可给 sim 长期库建 embedding，复用 livingmemory 同款 retriever 做附加联想深取。
- **硬边界**：sim 绝不把生活事件写进 livingmemory 的向量库；两域数据独立、可各自清理。

## 5. 行动循环细节（Act / Plan / World / 外联）

### 5.1 闲置触发 & 行动 roll

- **信号源**：`message_collect` → `lastActivity[presetId]=now`。不轮询；每次 roll 完若距 lastActivity < `idleThreshold`（默认 30min）则把下次 roll 推到 `lastActivity+idleThreshold`（人来了让世界让位）。落在 `quietHours`（默认 22:00–08:00）内的推到醒来。
- **与日程表协同**：wake 的实际时机不只看 next_delay，而是 `min(next_delay, 当前块尾, 下一个定时安排)`——见 §5.12。
- **行动 roll 输入**（静态在前，利于 prompt cache）：
  1. 角色 canon：人设摘要 + worldbook 查询（`selectEntries` 拿相关条目）
  2. `life-state`（§5.2）
  3. WorldContext（时段/天气/季节/地点，§5.4world）
  4. **当前日程块**：DailyPlan 里此刻计划在做的 activity/location（§5.3）
  5. **当前可发生事件类型集**：EventRegistry 按 WorldContext 筛出
  6. **短期记忆**：`recent(presetId,n)` 直取最近若干条
  7. **相关长期记忆**：按当前 thread/entity **key 直取**（非向量）
  8. 主动外联沉默状态（未回时长/连续未回数）→ 供 want_to_share
  9. 指令：先列 3–5 候选 beat（list-then-roll），程序端随机抽 1
- **行动 roll 输出（JSON schema 约束）**：
```jsonc
{
  "candidates": ["去库房清点刀装","和膝丸在檐下喝茶","练习时被本丸的猫绊了一下", ...],
  "chosen_index": 2,             // 程序端最终抽样，可按 thread 相关度加权
  "event": {
    "title": "练习时被猫绊了一下", "narrative": "≤150字第一人称 beat",
    "event_type": "角色互动",      // 取自当前可发生类型集
    "location": "本丸·练习场", "participants": ["本丸的猫"],
    "mood": "好笑/无奈", "duration_minutes": 40,
    "importance": 0.3,            // 仅用于沉淀/遗忘排序，不做外联门槛
    "threads_touched": ["与猫的日常"], "type": "context"
  },
  "plan_adherence": "deviated",   // followed | deviated | interrupted | free
  "replan_hint": "猫把练习搅了，这块提前收",
  "want_to_share": {              // 人设主导（§5.7），折进本次调用，零额外成本
    "decision": "no",             // now | later | no
    "target": "审神者",
    "reason": "这点小事不值当特意说",
    "draft": "",                  // now 时草稿（分享真事，禁挽留/情感勒索）
    "thought": "下次见面提一句野花"  // later 时 → 入 ThoughtBuffer（不排闹钟）
  },
  "next_state": { /* 更新后 life-state */ },
  "next_delay_minutes": 55
}
```
- **护栏**（§6.3 详）：schema 框结构；narrative 仅单 beat 自由；不违反 worldbook 硬事实；`event_type` 取自动态可发生集；不落硬禁止清单（新命名角色/重大伤亡）；出本丸走许可制（§5.4d）。`fallbackToTemplate` 随机池兜底。

### 5.2 life-state 状态对象（防漂移锚）

始终回喂便宜模型的小状态块，存 `life_sim_state`：
```jsonc
{
  "presetId": "髭切-通用版（Character）",
  "location": "本丸·檐下", "current_activity": "擦拭刀身",
  "mood": "慵懒",                 // 随事件波动（额外功能 §7）
  "open_threads": [ {"id":"膝丸约练","desc":"答应明天对练","due":"明日"} ],
  "recent_event_ids": ["evt_...","evt_..."],
  "updatedAt": 1719...
}
```
每次 roll 后由 `next_state` 覆盖更新；同时注入对话（`{life_state}`）让 bot 知道"此刻在做什么"。

### 5.3 日程：自著常规日程 + 被安排 + 每日计划 + 治理三层级

**(a) 常规日程 RoutineTemplate（"每天常规做什么"）—— bot 自己主设计**：
```jsonc
{ "presetId":"...", "authoredBy":"self", "revisedAt":1719...,
  "weekly": { "default": [
    {"block":"清晨","activity":"起身、擦拭刀身","location":"自室"},
    {"block":"上午","activity":"练习场对练","location":"练习场"},
    {"block":"午后","activity":"巡视本丸 / 看书","location":"庭院"},
    {"block":"黄昏","activity":"泡茶、发呆","location":"檐下"},
    {"block":"夜","activity":"夜巡","location":"本丸各处"} ] } }
```
- **自著**：`RoutineAuthor` 用便宜模型从 **人设 + 近期生活记忆（实际爱做什么）+ 可选人给种子** 生成/修订；定期（`routineReviseEvery`）或生活重心变化时自修订（慵懒的髭切会把练习排少、发呆排多）。`authoredBy` 记 provenance。

**(b) 被安排队列 AssignmentQueue（"被安排做什么"）** `life_sim_assignment`：
```
id, presetId, desc, dueDay?, dueBlock?, source(约定|审神者|近侍|主控), assignedBy, status, threadId?
```
来源：①对话/互聊约定（open_threads 带 due）②审神者直接吩咐 ③近侍/主控（多 bot）。

**(c) 每日计划 DailyPlan** `life_sim_plan`：blocks 带 `start/activity/location/source(routine|assigned|improvised)/assignedBy/status(pending|done|skipped|interrupted)`。唤醒时（`planRegenAt`）由 DailyPlanner 用便宜模型（或纯程序合并+轻润色）= 常规日程 + 到期 Assignment + 昨日摘要 + 人设 生成。

**与 roll 的关系**：EventRoller 先查"当前块计划在做什么"，事件默认执行计划、或偏离、或块间自由（`plan_adherence` 四态）。**计划是主干，事件是纹理**——大幅降漂移、让世界连贯。日程块与 wake-up 自循环怎么合成一个节拍，见 §5.12。
**重规划**：`plan_adherence=interrupted`（雨天取消练习/审神者临时安排/别的 agent 来访）→ `DailyPlanner.replan` 从该点重生成，旧块标 interrupted/skipped。
**闭环价值**：真人「明天陪我做 X」/ 互聊「约了对练」→ Assignment → 明天真进模拟日程 → 下次「今天照你说的做了」。**日程把对话·事件·记忆串成因果链。**

**(e) 治理三层级（自己 → 近侍 → 审神者）**（和刀剑乱舞设定一致）：
- **① 自己**：自著常规日程、对被安排项按人设接受/协商/婉拒。
- **② 近侍 bot**（`kinjiPreset`，可轮值，**世界内角色**）：本丸总管，**审核并安排他人日程**——协调跨 bot 冲突、编排共同活动、汇总"本日の予定"，产出跨 agent 的"安排意图"。**这些意图由世界外的毛线球进程机械执行**（§3.3）；近侍只管世界内的提议/审核，不等于协调进程。
- **③ 审神者**：终审权，经 console 检查/修改任何日程、换近侍。
- 审批流：自著/自排 → 近侍审核编排（跨 bot）→ 审神者随时覆盖；冲突/越权近侍先裁、审神者兜底。

### 5.4 WorldContext + 动态事件类型 + 连续性 + 出本丸许可

**(a) WorldContext** `life_sim_world`（②阶段为**本 agent 私有**、放本地 DB；①阶段**提升为跨进程共享**、放本丸共享存储，所有 agent 读同一份，§3.3）：
```jsonc
{ "clock":1719705600, "timeOfDay":"午后", "season":"初夏", "weather":"晴",
  "locations":["本丸·主屋","练习场","库房","庭院","檐下","厨房"],
  "externalLocations":["城下町","近所","遠征先·山道"], "updatedAt":1719... }
```
- weather 每 `weatherTickHours`（默认 6h）按**马尔可夫相邻表**推进（晴→多云→雨→阴），所有 agent 共享同一片天；`weatherSource: internal|api`。时钟 = 现实时间（§5.11）。

**(b) 事件类型注册表（数据驱动、可增减、热重载）** `event-types.json`：条目 `{type, conditions{timeOfDay?/weather?/season?/location?/requiresOtherPresent?}, weight}`；roll 前按 WorldContext 筛出可发生集喂模型。

**(c) 时空连续性约束**（防"5 分钟前在纽约、一小时后到日本"）：时钟单调；地点合法（本丸内 ∪ 已许可外部）；移动花合理时间（不瞬移）；天气仅相邻转移（禁跳变）；季节随真实日期。约束只管"物理可能"，不限"人设驱动选择"。**天气归本丸自己**：clock=真实时间时，本丸天气默认是**本丸的天**（内部演化），bot 不宣称=审神者同城天，避免"你那晴天、它说下雨"的错位；想同城则 `weatherSource: api` 绑定地点。时间矛盾统一由 §5.11 处理。

**(d) 出本丸 = 许可制 + 外部世界书（不硬禁）**：预授权 `externalLocations` 白名单可直接去；白名单外 → bot 产出"想去做 X"诉求经 want_to_share 请审神者许可，获批才发生（MVP 先只用白名单）；外部世界书（§5.5 的一册）补外部 canon；出/归花路程时间。硬禁止只剩**新命名角色/重大伤亡**。

### 5.5 世界书：可扩展多兼容 canon 层（加类目不改架构）

- **(a) 类目开放**：`category` 自由字符串，代码永不硬编码类目常量，config 里按字符串引用；加类目 = 加条目、零代码。
- **(b) 前向兼容 schema**：核心字段沿用 worldbook + 开放扩展袋 `ext{}`（任意 KV）；loader 宽容，未知字段保留不报错。未来加 `region/rating/validSeason/…` 塞 `ext`，老代码不受影响。
- **(c) 多格式 adapter**：`st`（复用 `convertStWorldbook`）/`koishi`/`typed` 归一到内部超集；新格式 = 加 adapter。
- **(d) 按用途分册 + 稳定查询**：`worldbooks:[{path,purpose}]`（persona/external/reference/未来任意）；只走 `query({purpose?,categories?,scanBuffer})→entries`。加册、加类目、加 ext 维度对 sim 全透明。
- 定性为对现有 worldbook 插件的**小幅前向兼容增强**（老条目不带 `ext` 也照跑）。

### 5.6 反思 Reflect（斯坦福）

- **触发**：累积重要度（近期事件 importance 之和过阈，仿斯坦福>150 的轻量版）或夜间固定跑。
- **过程**：取最近若干条 → LLM 抽"几个最值得追问的高层问题" → 每问按 key/相关记忆检索 → 合成"高层洞见"（关系变化、自我认知、习惯发现）→ 写回**长期**（可再被反思，成树）。
- **产物用途**：喂给 RoutineAuthor（习惯养成）、关系档案（关系演化）、下次 roll（自我一致）。

### 5.7 想法缓存区 ThoughtBuffer（心事簿）

`life_sim_thought`：`{id,presetId,content,target(审神者|presetId|self),origin,urgency,status(pending|surfaced|dropped|merged),relatedThreadId,createdAt,revisedAt}`。
- **存**：`want_to_share=later` 或"想说但不急"的念头入库（**取代 later+定时重查**：parking，不排闹钟）。
- **调**：下次和 target 接触时（真人来聊 / 经 peer 对话遇到那 bot），相关 pending 注入 `{pending_thoughts}` → "对了，之前一直想跟你说…"。
- **改**：更新/合并/作废（事已自解则 drop）；夜间随沉淀整理陈旧项。
- **分界**：livingmemory=过去事实、ThoughtBuffer=想说未说的意图、open_threads=进行中处境；说出即 `surfaced` 并沉淀成记忆。

### 5.8 主动外联（人设主导 + 两道可开关硬熔断）

> 研究警报：HBS 审计 1200 次告别 43% 用情感操控被骂粘人。**主动性优化参与度 = 必然读作粘人。**

- **决策交给人设**：`want_to_share.decision`（now/later/no）由角色性格决定想不想说、何时说、没回是否再发（沉默状态当输入喂进去）；无 importance 门槛。system 禁六类操控话术，draft 只分享真事。`later` → ThoughtBuffer。
- **两道可开关硬熔断**（默认开，可关成全人设驱动）：① `quietHoursEnabled` 安静时段不发；② `dailyCapEnabled` 每日上限（防写崩人设刷屏）。它们不是重要度门槛，是"不半夜发、一天别发二十条"的物理礼貌。
- **发送**：经 relay 吃其限流/白名单/防环；`proactiveEnabled` 可一键关（退化为纯被动浮现，最安全）。

### 5.9 持久化调度（重启不失忆）

`life_sim_task`：`{id,presetId,fireAt,type(roll|block|consolidate|prune|plan|reflect),status,payload}`（`block` = 日程块边界硬唤醒，§5.12）。`ready` 时**从"现在"续排下一次 wake**（过期任务丢弃或单次收尾，不补档——时间就是现实时间，关机那段就是"不在"，§5.11）+ 未来的 `setTimeout`；roll 完按 §5.12 的 `min(next_delay, 块尾, 定时安排)` 插下条；`dispose` 清内存 timer、DB 保留；并发安全见 §5.11（单活动锁）。

### 5.10 夜间「睡觉」

到 `sleepHour`：短→长沉淀（§4.3）+ 标记计划 done/skipped、未完成 assigned 顺延/放弃 + 生成次日计划 + 整理 ThoughtBuffer + 反思。

### 5.11 时间 = 现实时间；要处理的是并发

**单一时钟到底**：本丸时间就是现实时间（锚 `timezone`），**不做时间线对账/补档**——进程在跑就在生活，进程关着就是"不在"（诚实的空档，不编造假日常）。时段/季节/节日全从真实时钟派生。天气归本丸自己（§5.4c），不宣称=同城。

**真正的难点是并发**：不像斯坦福小镇是单进程 tick 步进，这里是**真实时间 + 多进程异步**——4 个 app + 毛线球 各自按真实时间独立触发，同一真实时刻可能同时动作、同时碰共享状态。故设计必须**并发安全**，而非依赖全局回合序。四条：

1. **共享状态单一属主**：毛线球侧的共享状态（共享 WorldContext / 在场登记 / 进行中 transcript / 收件箱）**只由毛线球读写**（单写者）；各 app 只提交意图/turn + 轮询收件箱，不多写者直抢 → 无竞态。（关系档案是各 bot **本地私有**，不涉跨进程竞态。同机若用共享 sqlite，另靠写锁 + 毛线球串行化兜底。）
2. **每 agent 单活动锁**：一个 agent 同一时刻只能处于 `roll / 被卷入互聊 / 正陪用户聊` 之一；忙时不被重复调度（扩展现有"同 preset 单 running roll 锁"）。**正陪用户时不 roll、不被拉去互聊**——顺带解决"生活事件画到正陪你聊天的时段"。
3. **碰撞不硬拒，走真人式社交解法**：A 醒来要找 B，毛线球查 B 状态——B 空则聊；**B 忙（陪用户/在别的互聊/忙活动）则按人设给社交结果**：
   - **加入**：若 B 正与 C 互聊（非私下陪用户），A 可打招呼后并入 → 多方互聊（A,B,C）。**单活动锁仍成立**：加入 = 进 B 那**唯一**活动，不新起一场。
   - **打招呼即走**：A 招呼一声、见 B 忙就走（可留一句余味）。
   - **回头再来**：A 记进 ThoughtBuffer / 改约（一条 Assignment），下次 wake-up 再试。
   - 选哪个由人设 + 情境定（便宜模型或简单启发）。这比"锁死 B、拒绝 C"像真人得多，也更 Smallville。
4. **共享资源原子分配**：地点/对方可用性等共享资源**原子更新、先占先得**（毛线球单写），避免两 agent 同一真实时刻都"占用练习场"、或都"在和对方对练"却只模拟了一场。

### 5.12 wake-up 自循环 × 日程表 协同

两个"节拍"要合成一个，否则自循环和日程各走各的：

- **日程 = 粗骨架**：DailyPlan 的 block 边界落在**真实时刻**（08:00 擦刀 / 10:00 和膝丸对练 / 14:00 远征），来自自著常规日程 + 被安排。提供"此刻该做什么"的锚 + **硬唤醒点**。
- **next_delay = 细纹理**：块**内**由模型自报的 `next_delay` 决定"同一活动里下一件小事多久后发生"。

**下一次 wake 时间 = 三者取最早**：
```
nextWake = min( now + next_delay,             // 块内纹理自节奏
                当前 block 的 end,              // 块边界(该切下一件事)
                下一个定时 Assignment 的 start ) // 如 10:00 对练
再夹: 纹理才受 lastActivity + idleThreshold 限；跳过 quietHours；陪用户时让位
```
- **next_delay 不得越过当前块尾**：若自报值会冲进下一块，**夹到块边界** → 下次 roll 正好在切块处、落进新活动（不会"午后了还在 roll 练习"）。
- **块 start 是硬唤醒点**：每个 block 的 start（尤其定时的被安排项）在持久调度里**各落一条 wake 任务**，到点必醒，不靠 next_delay 撞上。

**"切块" wake 时做什么**：
- 把 life-state 切到新块的 activity/location（推进"此刻在做什么"）。
- 新块是**被安排的互动**（如"和膝丸对练"）→ 产出"去找膝丸"意图交毛线球（§3.3，发起统一在 wake-up，正好接上）。
- 新块是**常规活动** → 就在新活动里开始 roll 纹理。
- 新块是**出本丸/远征**等长活动 → 进 ongoing 状态，跨多次 wake 持续（§5.11 时长约束）。

**陪用户占用时（单活动锁 §5.11）**：定时块/互动到点却正陪用户聊 → 该块**让位**（标 interrupted/skipped）：
- 被安排的互动 → "回头再来"（改约成新 Assignment / 记 ThoughtBuffer）；
- 常规块 → 直接跳过，聊完从当前真实时刻所在的块续上。
- 这不是 bug，是"陪你 > 自己过日子"的正确取舍。

**与真实时间/重启**：块边界是真实时间戳；重启后 Scheduler 从"现在"看当前落在哪个块、续排（§5.9/5.11）——错过的过去块就过去了，不补。

**每日边界**：sleepHour 后进 quiet，末个夜间块结束 → 下一次 wake 直接跳到**次日晨起第一个块**（跳过安静窗口）；起床时 DailyPlanner 已备好当日 blocks。

### 5.13 陪主人 ↔ 自己过日子 的交接（对话中断怎么办）

"陪你 > 过日子"的**交界**最易穿帮，三个顶层状态（单活动锁 §5.11 的具体化）：`WITH_USER`（陪主人）/ `LINGERING`（对话刚静下来、还没走）/ `LIVING`（自己过日子）。由 **PresenceState**（IdleWatcher 扩展）管理。

**① 主人来消息、bot 正在做某事 → 无条件让位（最高优先）**：
- 打断当前活动（标 interrupted，回头续做 / 按当前真实块走，§5.12）。
- **回复带"正在做的事"的语境**：life-state 已注入（"正擦刀呢"）→ bot 自然在情境里回应（"正擦着刀，怎么了？"），不凭空冒出。回复走既有 chatluna-character 管线，life-sim 只让路。

**② 主人聊两句忽然消失 → 不立刻回去过日子（那样像你一停就撇下你），进 LINGERING**：
- 用户静下来 → 排一个**短 `lingerWindowMin`（默认 4min）后的评估 wake**（复用同一 wake-up 心跳，不另设机制）。
- 到点 bot 按**人设**判断（把"**已追问 N 次 / 静默 M 分钟**"当输入喂进去）：
  - **追问**（细心的角色 → "还在吗？/接着刚才的话头"）；
  - 或**自觉"可以挂着等回复"** → 释放 WITH_USER、回 LIVING，把这段对话标成"刚聊过、悬着"。
  - **要不要再追、连环不连环，全交 bot 自判**：膝丸自然收手、髭切懒得追、黏人角色可能多问一句——不设死上限。runaway 由 §5.8 的两道可开关熔断（安静时段 / 每日上限）当最终兜底（追问复用 ProactiveBridge 发送，天然吃这两道）。

**③ 回到 LIVING → 无缝衔接**：
- 续做被打断的活动 / 或按 §5.12 进当前真实块。
- **对话线挂 top-of-mind**（进 recent/life-state）→ 主人再开口自然接上（"回来啦～刚我把刀擦完了" / 接回原话题），不从头。
- 没等到回复也不干等：超 `lingerWindowMin` + bot 自判 → 正常过日子，对话线转成一条 recent 记忆（"和主人聊到一半断了"）。
- **回复永远能再抢占**：任何时刻主人新消息 → 立即回 WITH_USER。

**与节奏协同**：LINGERING **不 roll** 生活事件；回 LIVING 后首个 roll 仍受 idleThreshold 从最后一条用户消息起算（§5.1）——别刚静下来就急着 roll。追问复用 ProactiveBridge 的发送 + 反操控底线（§5.8），但作为"续活对话"触发、比冷启动外联更自然。

## 6. 可行性与坑

### 6.1 成本账
- 醒着约 14h，事件间隔多 40–120min → **8–20 次 roll/天/agent**；单次输入 ~1.5–2.5k、输出 ~300–500 token。
- 便宜模型：默认**本地 ollama 免费**；云端 gemini-flash/siliconflow 约每天几分钱/agent。夜间沉淀 1 次 ~3–5k。
- 省钱：静态前缀吃 prompt cache；事件驱动非轮询（省 10–20×）；FrugalGPT 级联（本地兜底，低置信升云端）；`dailyRollCap`。多 agent 全开云端也就每天一两毛，本地近乎免费。

### 6.2 调度可靠性
- koishi 定时器重启即丢、无 cron → §5.9 DB 落库+启动扫描是**必须**。过期任务单次补跑，不堆积雪崩。

### 6.3 漂移与失控（半自由叙事如何不崩）
schema 约束 + life-state 锚 + 每次重注人设 canon + worldbook 硬护栏 + 动态类型集/硬禁止清单 + 计划当主干 + dry-run（只 log 不写库调参）+（后置）canon 校验不合规重 roll。

### 6.4 与真人聊天的衔接
sim-memory 多变量**直接注入**，和 livingmemory 并排 → 自然不突兀。关键不变量：sim 记忆与真人记忆**格式同构**。

### 6.5 主动外联的雷区
见 §5.8：模拟(常跑)与外联(极少)分离 + 人设主导 + 禁操控话术 + 两道可开关熔断。

### 6.6 多 agent 互聊
见 §3.3：各 bot 用自己语境 + `peerModel` 真实生成 turn（不跑用户窗口）；**收尾归 bot 自己**；毛线球只路由/持权威 transcript/硬安全帽熔断（`botChatMaxTurns`/成本）防跑不停；防 echo 靠各自真人设；`botChatDailyCap` 限频。

| **纪念日/回忆** | 旧事在周年浮现 | 日期匹配长期重要事件 | 长期记忆 `date` key 直取 | **P3** |

（这些都**只挂在既有核心接口上**：事件走 EventRegistry/roll、记忆走长短期两层、日程走 Planner、主动走 ProactiveBridge——不新增架构。）

## 9. 分阶段 MVP

> 主轴：**P1/P2 = ② 日常生成层（进程内，装进单个 app 即可）；P3 = ① 协作层（跨进程共享层，新基础设施）。** 已有的多 agent 架构本身不用动。

### Phase 1 — ② 日常生成层落在单 agent（先做，最划算）
装进一个 app（如 髭切）进程内，用它自己的 DB，独立验证"不聊天世界也在长 + 可引用一致"，80% 价值在此：
1. **持久化 Scheduler**（DB+ready 扫描+dispose）——硬地基。
2. **IdleWatcher / PresenceState**（订阅 message_collect + WITH_USER/LINGERING/LIVING 交接：优先回复、追问、自判去留、无缝衔接，§5.13）。
3. **WorldContext + EventRegistry**（含连续性约束）。
4. **RoutineAuthor + DailyPlanner + DailyPlan + wake×日程协同**（bot 自著常规日程 → 每日计划当主干；块边界当硬唤醒点、next_delay 夹到块尾，§5.12；重规划/对话安排留 P2）。
5. **审神者查看/修改日程**（console/指令）。
6. **EventRoller（Act）**（便宜模型 + schema + life-state 锚 + worldbook + 当前日程块 + 动态类型集 + list-then-roll；fallback 兜底）。
7. **ShortTermMemory**（直取式短期工作集 + 各表）。
8. **NightConsolidator**（**短→长沉淀**：聚类 keep/update/merge/archive 进 LongTermMemory；标记计划 + 生成次日计划）。
9. **ThoughtBuffer**（对审神者：store + 下次来聊 surface + revise）。
10. **心情轻量**（mood 字段随事件波动）。
11. **最小 ProactiveBridge**（人设 want_to_share 主导 + 可开关熔断；later→ThoughtBuffer）。
12. **PromptProvider** 注入 `{recent_life}/{life_state}/{today_plan}/{pending_thoughts}`。
13. dry-run + debug + 本地模型默认。

**验收**：开一两天后，`life_sim_event` 有贴合日程主干、随天气/时段变化的连续事件流；长期库有夜间沉淀的条目；下次聊天它能自然、**一致**地提起这两天做的事（你追问对得上）；主动消息符合人设节奏且不粘人。

### Phase 2 — 打磨与深化
- **反思 Reflector**（累积重要度触发，写回长期）。
- **长期记忆压缩/遗忘**（dream 式 + 引用计数）+ 可选给长期建 embedding 做联想深取。
- **周报/日记生成**、**突发事件**、**节日/季节事件**。
- 日程闭环：AssignmentQueue 录入「对话约定/审神者吩咐」+ 重规划；出本丸"请求许可"流。
- canon 校验（不合规重 roll）。

### Phase 3 — ① 协作层（跨进程共享，新基础设施）
- **毛线球协调/路由进程**（世界外，headless 第 5 进程，**不生成内容**）：托管共享存储（共享 WorldContext + 在场登记 + 进行中 transcript + 交互队列 + 每 agent 收件箱）+ 撮合/社交仲裁 + 路由轮次 + 硬安全帽熔断 + 分发（§3.3）。
- **各 agent 端 PeerConversant**：用**自己完整语境 + `peerModel`** 生成本方 turn（peer 路径，不跑用户窗口）；**自己决定收尾**；会后从权威 transcript 抽自己的记忆 + 更新**本地自己视角的关系档案**。
- **发起统一在 agent wake-up**（不另设毛线球时钟）：wake-up 产"去找X"意图 → 毛线球仲裁社交结果 → 路由。默认 2 方；碰撞可**加入成多方**（§5.11 点 3）。
- **关系演化**（本地关系档案随交互漂移）。
- **近侍（世界内角色）产出安排意图 → 毛线球执行**（两者解耦）。
- `visibleVia: relay` 围观升级（真发消息/群聊）。

## 10. 风险登记

| 风险 | 等级 | 缓解 |
|---|---|---|
| 靠向量抓低价值日常、抓不到 | 高（已规避） | **记忆改直取式两层**：recency 短期直取 + key 长期直取，向量仅可选联想（§4.4） |
| 重启丢闹钟、本丸停摆 | 高 | DB 落库 + ready 扫描补跑（§5.9）——P1 第一件事 |
| 并发（多进程真实时间异步、抢共享状态/资源） | 中高 | 共享状态单一属主(毛线球单写) + 每 agent 单活动锁 + 碰撞走社交解法(加入/打招呼即走/回头再来) + 共享资源原子分配（§5.11） |
| 便宜模型 roll 崩人设 | 中高 | schema + life-state 锚 + worldbook + 动态类型集/硬禁止 + 计划主干 + dry-run（§6.3） |
| 主动消息读作粘人 | 中高 | 人设主导 want_to_share（含没回是否再发）+ 禁操控话术 + 两道可开关熔断（§5.8） |
| 世界跳变（瞬移/天气突变/乱跑外地） | 中 | 时空连续性约束 + 出本丸许可制（§5.4c/d） |
| 日程太死板、像打卡 | 中 | bot 自著日程 + 计划只当主干、事件可偏离/打断/自由；重规划（§5.3） |
| 对话中断显得冷/被撇下 | 中 | WITH_USER/LINGERING/LIVING 交接：回复优先 + 人设自判是否追问(喂已追N次/静默M分,不设死上限,§5.8熔断兜底) + 自判去留 + 回来无缝衔接（§5.13）|
| 与 livingmemory 打架 | 中 | 分域：sim 不写它向量、只借鉴 dream 思路；重要生活事件靠对话自然回流（§4.6） |
| 多 agent 互聊跑飞/成本爆炸 | 高（P3） | 收尾归 bot 自己（自然结束）+ 毛线球硬安全帽熔断兜底 + 各自真人设防 echo + `botChatDailyCap`（§3.3/6.6） |
| 关系脱离 bot 产出（旧集中式病灶） | 已规避 | 毛线球不生成；各 bot 用自己完整语境真实生成 turn、自己抽记忆、本地存自己视角关系档案（§3.3/4.2） |
| 跨进程一致性 | 中（P3） | 毛线球持**单一权威 transcript**，双方从同源抽取 → 事实对齐、视角分化；本地记忆权威（§3.3） |
| bot↔bot 污染对话域 | 中（P3） | peer 路径独立于用户窗口，不进 chatluna-character buffer/livingmemory extraction（§3.3） |
| 近侍越权 / 抢资源 | 中（P3） | 三层级：近侍先裁、审神者终审覆盖（§5.3e）；近侍仅提议、毛线球执行 |

## 11. 参考（人机恋 / agent 研究）

- **Generative Agents（Park 2023）**：memory stream + 检索三因子（recency·importance·relevance，等权 min-max 归一）+ reflection（重要度和>150 触发，100 条→3 问→每问 5 洞见，成树）+ top-down 规划（日→时→5-15min，反应即重规划）+ agent 对话（共处→取关系记忆→轮替至结束→记回流）。**本设计把三因子拆开分工、把空间共处换成毛线球（协调进程）撮合。**
- **AI Town（a16z）**：recency 衰减 0.99/h，importance 0–9，等权。
- **Letta sleep-time**：只读 primary + 后台 rewriter；分层 cadence（session 去重/session-end 轻整理/weekly 重组）；reference-count 比 age 更重要；别过度归并丢粒度。
- **Mem0**：ADD/UPDATE/DELETE/NOOP 冲突解决。
- **ComPeer**：事件检测 + 晨间反思 + 随机阈值门 + 3h 无回复抑制。
- **Nomi/Kindroid**：闲置触发 + 指数退避 + 硬安静时段 22:00–08:00 + "基于真在想/做的事"非脚本。
- **HBS 告别审计**：43% 情感操控 → 主动外联最大雷区。
- **Ian Bicking**：list-then-roll（LLM 列表→程序抽样）；分层世界状态。
- **AutoGen**：MaxMessage/TERMINATE/TokenUsage 终止；echoing 论文：每轮重注角色卡降 40–60%。
- **FrugalGPT**：便宜优先级联，最高 98% 降本。
- **livingmemory（本仓依赖）**：`living_memory_entry`（type/status/importance0-1/embedding）；`runDream` 两阶段聚类 keep/update/merge/archive；vector+importance 检索。**sim-memory 复用其整理思路、各管各域。**
