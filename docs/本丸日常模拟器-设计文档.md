# 本丸日常模拟器（chatluna-life-sim）设计文档

> 状态：设计阶段（不改运行代码）
> 日期：2026-06-30
> 作者：妖祀 / Claude
> 一句话：让 bot 在没人聊天时，用便宜模型「自己过日子」——roll 事件、自定闹钟、当天攒成事件表，夜里用主模型整理归档进 livingmemory，于是「不聊天，世界也在长」。

---

## 0. TL;DR

- **核心机制**：闲置 30min（可配）→ 便宜模型在「人设/世界观/当前状态」约束下 roll 一个事件 + 结果 → 模型自报「下一件事多久后发生」→ 落库当闹钟 → 到点再 roll（自循环）。当天攒成事件表，夜里「睡觉」时主模型整理归档进 livingmemory，并触发 dream 聚类。每周清理低重要度残留。
- **站位**：模拟器是 livingmemory 的**写入端生产者**；事件靠**现有记忆注入通路**（`{living_memory}`）在下次对话自然浮现——衔接问题已被现有架构解决，无需新注入路径。
- **关键护栏**：JSON schema 框结构 + 始终回喂的 `life-state` 状态对象 + 每次 roll 重注人设 canon（查 worldbook）→ 让「半自由叙事」既生动又不崩人设。
- **动态世界**：事件类型不是静态白名单，而是 **WorldContext（时段/天气/季节/地点）筛出的动态可发生集**（可增减、热重载），天气会自演化——世界随时间/天气/地点真的在变（§4.9）。
- **主动外联**：**无 importance 门槛**，发不发、何时发由 roll 输出的 `want_to_share` **交给人设判断**（折进同一次调用、零额外成本）；只保留三道与重要度无关的**机械底线**（安静时段/日上限/无回复抑制）当纯熔断（§5.5）。
- **MVP**：单 bot 生活自循环 + 最小主动发消息。默认便宜模型本地 ollama（可配云端），调用逻辑照搬 photo 插件读 `ctx.chatluna.createChatModel`。
- **不在 MVP**：bot 间自控互聊、高维主控（留三期）。互聊**默认走「互聊代理」ChatProxy**（便宜模型内部模拟、产物=共享记忆），**不用 relay 跑主模型窗口**；relay 仅在想让互聊可见时用（§6.0）。

---

## 1. 价值判断

### 1.1 它真正解决什么

当前 bot 的「主动性」与「连续生活感」完全依赖 relay 被动转发——**不聊就不长，世界是静止的**。本设计给 livingmemory 喂一条**不依赖真人输入的记忆增长流**，使下次对话能自然带出「我这两天……」。这对「像真人」的贡献是**结构性的**，不是锦上添花：真人之所以像真人，正因为 ta 在你不在时也在经历事情。

三个具体收益：
1. **拟真感**：「世界在长」是 AI 伴侣类产品公认的高价值幻觉（Nomi「消息基于它真在想/做的事」、Kindroid「mini-stories from recent conversation」）。
2. **降低对 relay 的依赖**：事件流 + 近期话题为未来的 bot 自控互聊提供素材，主控不必每次靠真人触发。
3. **主动性的安全形态**：bot 有故事可讲（被动浮现） > bot 追着你发消息（主动 push）。后者是雷区（见 §5.5）。

### 1.2 对「像真人」目标的贡献度

高，且**站位极好**——几乎所有零件都有现成地基（§2），主要新增工作是「持久化调度 + 受约束的事件生成 + 夜间归档编排」，而非从零造记忆系统。

### 1.3 有没有更简单的替代

| 替代方案 | 成本 | 漂移 | 「世界在长」 | 结论 |
|---|---|---|---|---|
| 纯随机事件池（不调模型，预写事件抽签） | 零 | 零 | 弱（重样、不长记忆） | 当**冷启动/降级兜底**，不能替代 |
| 只在你聊天时即兴编「这几天发生的事」 | 低 | 中 | 假的（每次现编、不连续） | 不连续，露馅 |
| **本设计（便宜模型 + 持久事件流 + 夜间归档）** | 低（便宜模型/本地免费） | 可控（§5.3） | 真 | 推荐 |

→ 纯随机池保留为**可切换的兜底挡位**（模型不可用 / 省钱模式），但默认走模型 roll。

## 2. 现成地基（站在哪些肩膀上）

| 能力 | 基础设施 | 关键接口 / 事实 |
|---|---|---|
| 记忆读写/重要度/归档 | **livingmemory**（第三方 npm，装在各 app node_modules） | `ctx.chatluna_living_memory`：`createMemory(scope,input)` / `listMemories(query)` / `updateMemory` / `getMemory` / `deleteMemory`；记录表 `living_memory_entry`，字段含 `type(identity/preference/fact/plan/context/other)`、`status(active/archived)`、`content`、`keywords[]`、`summary`、`sentiment`、`importance(0-1)`、`sourceConversationId`；`createScope(conversationId,presetId,userId?,channelId?,options?)→MemoryScope` |
| 夜间整理（dream） | livingmemory | `runDream(presetId)`：聚类 active 记忆 → LLM 决策 keep/update/merge/archive；**只能手动经 console 按钮触发，无自动定时**——正是我们的夜间钩子机会 |
| 跨窗口记忆互通 | **scene-rules**（自研） | 把多个 presetId（私聊/语音/本丸版）经 `aliases` 映射到**同一记忆池**；包裹 `createScope`/`resolvePresetId`。→ 模拟器写一次，所有窗口可读 |
| 人设/世界观护栏 | **worldbook**（自研，feat-worldbook worktree） | 关键词触发注入；导出纯函数 `selectEntries/entryActivates/matchKey/buildScanBuffer`；条目 `{keys,secondaryKeys,logic,constant,content,category,order}`；有「刀男人设/审神者」分类 |
| bot 间通信（二期/三期） | **relay v2**（自研） | `[[relay:别名\|话\|图=desc\|nsfw]]` 标记，经 `chatluna_character/raw-response` 带外执行 `bot.sendPrivateMessage`；五层护栏（trigger 白名单 / recipient 白名单 / 好友 / 限流 minIntervalSec+dailyLimit / 防环 myBots）；`pushToBuffer(key,content,bot)` 双向注入 `chatluna_character.getMessages(key)`；`relayKeyOf(session)` |
| 闲置/活动信号 | **chatluna-character**（外部） | 事件 `chatluna_character/message_collect`（有人说话）、`raw-response`、`clear-chat-history`；buffer 是内存态 |
| 持久化范式 | **character-buffer-backup**（自研） | DB 表 + `ctx.on('ready')` 启动恢复 + `ctx.on('dispose')` flush 的范式可直接抄；`ctx.database.extend(table,schema,opts)` |
| 调度原语 | koishi（@cordisjs/timer） | `ctx.setTimeout/setInterval/sleep`（返回 disposer，**内存态，重启即丢**）；**未装 cron**。⚠️ 必须自建 DB 表 + 启动扫描 |
| 模型调用 | chatluna | `await ctx.chatluna.createChatModel(platform, name)` → `.value`（可能是 ComputedRef）→ `.invoke(lcMessages,{signal})`；用 LangChain `SystemMessage/HumanMessage/AIMessage`；便宜模型现成：**ollama 本地免费**、siliconflow、openai-like gemini-flash。**调用逻辑照搬 `koishi-plugin-photo/model.js`** |

**两条最关键发现**：
1. **「自定闹钟」必须落库**——koishi 定时器重启即丢，否则一次重启整个本丸失忆停摆。这是 MVP 的隐形硬地基。
2. **事件不需要新注入通路**——写进 livingmemory 后，现有 `{living_memory}` 注入下次聊天自动带出，衔接问题已解决。模拟器只是**写入端**。

## 3. 架构总览

```
[Idle Watcher] 监听 message_collect → 闲置 > idleThreshold(默认30min,可配) → enqueue wake
      │
[Scheduler·持久化] DB: life_sim_task{id,presetId,fireAt,type,status,payload}
      │   ctx.on('ready') 扫过期任务 → 执行；ctx.setTimeout 排下一个
      │   roll 完模型自报 next_delay → 再插一条任务（自循环）
      ▼
[WorldContext 本丸级] 时钟/天气(自演化)/季节/地点 ──┐(跨角色共享)
[EventRegistry 动态] 按上下文筛出"可发生类型集" ──┤
                                                    ▼
[Event Roller·便宜模型] 输入: 人设canon(worldbook+八字) + life-state + WorldContext + 可发生类型集 + 近期记忆 + 近期话题
      │   list-then-roll: 列 N 个候选 beat → 程序随机抽一个
      │   输出(JSON schema): event{event_type,...} + want_to_share(人设判断) + 更新后 life-state + next_delay
      ├──→ [Life-State Store] DB: life_sim_state{presetId,stateJson}  (始终回喂=防漂移锚)
      ├──→ [Event Log] DB: life_sim_event{...}  (当天事件缓存表)
      └──→ [Proactive Bridge] want_to_share.decision + 机械底线 → 极少数才发
      ▼  (夜间 sleepHour 触发)
[Night Consolidator·主模型] 读当天事件 → createMemory 写 consolidated 记忆
      │                      → runDream(presetId) 聚类整理 → 标记 events consolidated
      ▼
[Weekly Pruner] 周清理低重要度未归档事件（长期库交给 livingmemory dream/archive 兜底）

注入: 既有 {living_memory} 自然带出已归档事件；新增 {life_state} 让 bot 知道"此刻我在做什么"

主动外联(MVP,人设主导): want_to_share=now/later/no → 撞机械底线(安静时段/日上限/无回复抑制)才真发
—— 三期 —— [高维主控] 发起+节流 → [ChatProxy] 便宜模型内部模拟互聊 → 产物=双方共享记忆(默认不经relay)
```

### 3.1 单元职责与边界（隔离设计）

| 单元 | 做什么 | 怎么用 | 依赖 |
|---|---|---|---|
| **IdleWatcher** | 跟踪每个 preset 的最后活动时间，闲置超阈值通知调度器 | 订阅 `message_collect`；`recordActivity(presetId)` / 检测闲置 | chatluna_character 事件 |
| **Scheduler** | 持久化定时：落库 + 启动扫描 + setTimeout | `scheduleTask(presetId,fireAt,type)` / `onReady 扫描` / `dispose 清 timer` | ctx.database, ctx.timer |
| **EventRoller** | 便宜模型生成受约束事件 | `roll(presetId)→{event,nextState,nextDelay}` | chatluna model, worldbook, livingmemory(读) |
| **LifeStateStore** | 存/取小状态对象（当前在哪/心情/未了线） | `get(presetId)` / `set(presetId,state)` | ctx.database |
| **WorldContext** | 本丸级世界状态（时钟/天气/季节/地点），跨角色共享 | `get()` / `tickWeather()` | ctx.database（可选 天气 API） |
| **EventRegistry** | 动态事件类型注册表，按上下文筛可发生集 | `available(worldCtx,lifeState)` | 文件/表，热重载 |
| **EventLog** | 当天事件缓存 | `append(presetId,event)` / `listDay(presetId,day)` / `markConsolidated` | ctx.database |
| **NightConsolidator** | 夜间主模型整理 → 写 livingmemory + dream | `consolidate(presetId,day)` | chatluna model, livingmemory(写) |
| **WeeklyPruner** | 周清理低重要度残留 | `prune(presetId)` | ctx.database, livingmemory |
| **PromptProvider** | 注入 `{life_state}` 变量 | `registerFunctionProvider('life_state',...)` | chatluna.promptRenderer |
| **ProactiveBridge** | 人设想发(want_to_share)且过机械底线才发 | `maybeReachOut(presetId,event,share)` | relay / bot.sendPrivateMessage |
| **ChatProxy**（三期） | 内部模拟 bot↔bot 互聊，产物=共享记忆 | `simulate(a,b,seed)→摘要` | 便宜模型（默认不经 relay） |
| **Orchestrator**（三期） | 跨 bot 发起+节流+话题种子 | `maybeStartChat()` | ChatProxy / relay(visible) |

每个单元可独立测试（纯逻辑：gate 判定、importance 计算、list-then-roll 抽样、调度到期判定都不依赖 IO）。

## 4. 核心机制·数据结构与接口

### 4.1 闲置触发

- **信号源**：`chatluna_character/message_collect`（任意人在该 session 说话）→ `lastActivityAt[presetId] = now`。
- **判定**：模拟器不轮询；而是每次 roll 结束时，若距 lastActivity < idleThreshold，则把下次 roll 推迟到 `lastActivity + idleThreshold`（人来了就让世界让位）。等价于「闲置满 30min 才允许下一次 roll」。
- **安静时段**：落在 `quietHours`（默认 22:00–08:00，可配，对齐 Nomi）内的任务推迟到醒来时刻——bot 也要「睡觉」。
- 阈值 `idleThreshold` 默认 30min（用户指定），可配；参考 ComPeer 5min / Kindroid ~20min，30min 偏保守、更不打扰。

### 4.2 事件 roll（便宜模型）+ list-then-roll

**输入 prompt 组装**（静态在前，便于 prompt cache）：
1. 角色 canon：preset 人设摘要 + 八字 + worldbook 查询结果（用 `selectEntries` 拿与当前 life-state 相关条目）
2. `life-state` 当前状态对象（§4.3）
3. **WorldContext** 当前本丸世界上下文（时段/天气/季节/可用地点，§4.9）
4. **当前可发生的事件类型集**：由事件类型注册表按 WorldContext 筛出（§4.9）
5. 近期长期记忆：`listMemories({presetId, ...})` 取最近 / 高重要度若干条
6. 近期聊天话题：从 character buffer 取最近 N 条摘要
7. 指令：先**列 3–5 个候选 beat**（list-then-roll，治重样），程序端随机抽 1 个展开

**输出（JSON schema 约束，治漂移）**：
```jsonc
{
  "candidates": ["去库房清点刀装", "和膝丸在檐下喝茶", "练习时被本丸的猫绊了一下", ...], // 列表
  "chosen_index": 2,            // 模型可建议；程序端最终随机抽样（可加权重要度）
  "event": {
    "title": "练习时被猫绊了一下",
    "narrative": "半自由叙事，一段 beat（≤150字），第一人称",
    "event_type": "角色互动",     // 必须取自"当前可发生类型集"（动态，§4.9）
    "location": "本丸·练习场",
    "participants": ["本丸的猫"],
    "mood": "好笑/无奈",          // 影响 sentiment
    "duration_minutes": 40,       // 这件事花多久 → 影响 next_delay 基线
    "importance": 0.3,            // 0-1 自评（仅供夜间归档/周清理排序，不做外联门槛）
    "threads_touched": ["与猫的日常"], // 关联的未了线
    "type": "context"            // 映射 livingmemory type
  },
  "want_to_share": {             // 人设主导的主动外联判断（§5.5）——折进这一次调用，零额外成本
    "decision": "no",            // "now" | "later" | "no"：由人设决定想不想说
    "delay_minutes": 0,          // decision=later 时，多久后再考虑
    "reason": "这点小事不值当特意去说",
    "draft": ""                  // decision=now 时的草稿（分享真发生的事，禁挽留/情感勒索话术）
  },
  "next_state": { /* 更新后的 life-state，见 4.3 */ },
  "next_delay_minutes": 55       // 自定闹钟：下一件事多久后发生
}
```

**抽样策略**：程序端从 `candidates` 抽样而非盲信 `chosen_index`——可均匀，也可按「与未了线相关度」加权。随机性来自程序，连贯性来自模型（Ian Bicking）。

**护栏**（§5.3 详述）：JSON schema 框结构；narrative 仅在单个 beat 内自由；不许违反 worldbook 硬事实；`event_type` 必须取自当前 WorldContext 筛出的**动态可发生类型集**（§4.9），且不落**硬禁止清单**（出本丸远行/新命名角色/重大伤亡等需用户拍板的剧情）。

### 4.3 life-state 状态对象（防漂移锚）

始终回喂给便宜模型的小状态块，存 `life_sim_state` 表：
```jsonc
{
  "presetId": "髭切-通用版（Character）",
  "location": "本丸·檐下",
  "current_activity": "擦拭刀身",
  "mood": "慵懒",
  "open_threads": [               // 未了的线，给连续性
    {"id":"猫", "desc":"最近总有只猫来蹭", "last_ts": 1719...},
    {"id":"膝丸约练", "desc":"答应膝丸明天对练", "due":"明日"}
  ],
  "recent_event_ids": ["evt_...","evt_..."],  // 最近几件，防自我重复
  "updatedAt": 1719...
}
```
每次 roll 后由模型产出 `next_state` 覆盖更新。它同时被注入对话（§4.6 `{life_state}`），让 bot 知道「此刻我在做什么」。

### 4.4 事件表缓存（当天）

`life_sim_event` 表：
```
id            string primary
presetId      string(255)  index
day           string(10)   // 'YYYY-MM-DD'（按 bot 时区/起床日切）
ts            timestamp
title         string
narrative     text
location      string
participants  json
mood          string
duration_min  integer
importance    double       // 0-1
threads       json
type          string       // livingmemory type
consolidated  boolean      // 是否已夜间归档
sourceModel   string       // 哪个模型 roll 的（审计/调参）
```

### 4.5 夜间「睡觉」整理归档（主模型）

触发：到 `sleepHour`（默认 23:00，可配；落在 quietHours 起点附近）时，调度器排一个 `type=consolidate` 任务。

`NightConsolidator.consolidate(presetId, day)`：
1. `EventLog.listDay(presetId, day)` 取当天未归档事件。
2. **主模型**一次调用：把当天事件流喂进去，产出「**少量、去重、第一人称**的长期记忆候选」（对齐 livingmemory 的 content 风格：≤100字、keywords≤12、不堆日期）。Letta sleep-time 思路：把「raw 流」转成「learned 记忆」，不是逐条搬运。
3. 对每条候选调 `livingmemory.createMemory(scope, input)`，`scope = createScope(syntheticConversationId, presetId)`，携带 `importance / type / keywords / sentiment / summary`。
   - `sourceConversationId` 用一个合成 ID（如 `life-sim:{day}`）以便区分来源、可回溯/可清理。
4. 调 `livingmemory.runDream(presetId)` 触发聚类整理（它会自动 keep/update/merge/archive，把今天的并入历史）。
5. `EventLog.markConsolidated(当天事件)`。

**为什么靠 createMemory + runDream 而不是自己写库**：复用 livingmemory 既有的去重/合并/归档与跨窗口注入，避免造第二套记忆系统、避免与 scene-rules 冲突。

### 4.6 注入：事件如何自然进入下次对话

- **被动浮现（主路径）**：归档进 livingmemory 后，现有 `{living_memory}` 注入在下次聊天自动带出——**零额外工作**。事件因此「自然」而非「突兀」，因为它和真人产生的记忆走同一条注入路径。
- **当前状态**：新增 `{life_state}` 函数 provider（仿 worldbook 的 `registerFunctionProvider`），把 life-state 渲染成短文本注入，让 bot 知道「此刻在做什么」。可选放进 worldbook 常驻条目，二选一。

### 4.7 重要度标记 + 每周清理

- **重要度**：roll 时模型自评 `importance(0-1)`（poignancy）。研究提示**recency 实证几乎无用、新颖度/内容量主导**——故周清理**不按时间**，按 importance + 是否触及未了线。
- **每周清理** `WeeklyPruner.prune(presetId)`：
  - 删除 `life_sim_event` 中 `importance < pruneThreshold(默认0.25)` 且 `consolidated=true` 的当周残留（已归档的就不必留原始事件）。
  - 长期库的精简交给 livingmemory 自己的 dream/archive（active→archived→物理删除二段式），不重复造轮子。
  - 参考 Letta：reference-count 比 age 更重要——被反复触及的 thread 不删。

### 4.8 持久化调度（重启不失忆）

`life_sim_task` 表：
```
id        string primary
presetId  string(255) index
fireAt    timestamp   index
type      string      // 'roll' | 'consolidate' | 'prune'
status    string      // 'pending' | 'running' | 'done' | 'failed'
payload   json
```
- `ctx.on('ready')`：`get({status:'pending'})` → 过期的立即补跑（或丢弃过老的，按策略）；未来的 `ctx.setTimeout(fire, fireAt-now)`。
- roll 完插入下一条 `roll` 任务（`fireAt = now + next_delay`，再夹 idle/quiet 调整）。
- `ctx.on('dispose')`：清掉内存 timer（disposer 存 Map），DB 任务保留 → 重启重排。
- 防重入：同 preset 同时只允许一个 running roll（参考 livingmemory dream 的 per-preset 锁）。

### 4.9 WorldContext + 动态事件类型注册表（事件随时间/天气/地点变化）

事件类型**不是静态白名单**，而是「当前世界上下文筛出的可发生集」。两层：

**(a) WorldContext — 本丸级、跨角色共享的世界状态**（一个本丸一份，所有 preset 读同一份）：
```jsonc
{
  "clock": 1719705600,          // 内部时钟：默认=真实时间，可配缩放（如 1 现实天=1 本丸天）
  "timeOfDay": "午后",          // 由 clock 派生：清晨/上午/午后/黄昏/夜/深夜
  "season": "初夏",             // 由真实日期派生
  "weather": "晴",              // 自演化（见下），或接真实天气 API（可配）
  "locations": ["本丸·主屋","练习场","库房","庭院","檐下","厨房"], // 可用地点集
  "updatedAt": 1719...
}
```
- **weather 演化**：轻量马尔可夫/随机游走，每隔 `weatherTickHours`（默认 6h）由**便宜模型或纯程序表**推进一步（晴→多云→雨→阴→晴…），避免每 roll 现编导致天气跳变。天气进 WorldContext 后所有 bot 共享同一片天。
- **可选真实数据**：`weatherSource: internal|api`，`clockScale` 可配。MVP 用 internal。

**(b) 事件类型注册表 — 数据驱动、可增减、可热重载**（像 worldbook 一样存文件/表，改了热加载）：
```jsonc
[
  { "type": "练习",     "conditions": { "timeOfDay": ["上午","午后"], "weather": ["晴","多云"] }, "weight": 3 },
  { "type": "檐下发呆", "conditions": { "weather": ["雨","阴"] }, "weight": 2 },
  { "type": "赏雪",     "conditions": { "season": ["冬"], "weather": ["雪"] }, "weight": 2 },
  { "type": "夜巡",     "conditions": { "timeOfDay": ["夜","深夜"] }, "weight": 1 },
  { "type": "角色互动", "conditions": { "requiresOtherPresent": true }, "weight": 2 },
  { "type": "思绪",     "conditions": {}, "weight": 1 }   // 无条件，任何时候可发生
]
```
- **筛选**：roll 前 `availableTypes = registry.filter(t => matchContext(t.conditions, worldContext, lifeState))`；把 `availableTypes`（含权重）喂给模型，`event_type` 必须取其一。
- **动态**：注册表可随时增删条目、改条件/权重，热重载；不同本丸/preset 可挂不同注册表。这就是「事件类型随时间/天气/地理动态变化」的落点。
- **硬禁止清单**独立于注册表存在（出本丸远行/新命名角色/重大伤亡…），是安全底线，不受动态增减影响。
- **纯逻辑可测**：`matchContext()`、weather 演化、筛选+加权抽样都不依赖 IO。

## 5. 可行性与坑

### 5.1 成本账

- **节奏**：醒着约 14h，闲置 30min 起步、事件间隔由 `next_delay`（多为 40–120min）决定 → 约 **8–20 次 roll/天/bot**。
- **单次 roll**：输入 ~1.5–2.5k token（人设+state+近期记忆+话题），输出 ~300–500 token。
- **便宜模型**：默认**本地 ollama = 免费**；云端 gemini-flash/siliconflow 约 **每天几分钱/bot**。
- **夜间归档**：1 次主模型调用，输入当天事件 ~3–5k token。
- **省钱杠杆**（研究背书）：静态前缀（人设卡）放最前吃 prompt cache；事件驱动而非轮询（省 10–20×）；FrugalGPT 级联（本地兜底，低置信再升云端）；每日 roll 上限 `dailyRollCap` 防失控。
- 四个 bot 全开，云端也就「每天一两毛」，本地则近乎免费。**成本不是阻碍**。

### 5.2 调度可靠性

- koishi 定时器**重启即丢**且未装 cron → §4.8 的 DB 落库 + 启动扫描是**必须**，不是可选。已有 buffer-backup 范式可抄。
- 过期任务策略：重启后对「该跑没跑」的 roll，做**单次补跑**（不要把堆积的全补，否则雪崩）——过老的直接顺延到 now+idle。

### 5.3 漂移与失控（半自由叙事如何不崩）

多层护栏，从强到弱：
1. **JSON schema 约束输出**（grammar/structured output，Outlines/XGrammar 思路）：地点/人物/重要度/时长走结构化字段，narrative 仅在单 beat 内自由。
2. **life-state 锚**：始终回喂当前状态对象（Letta core memory / Character-LLM 实证：外置状态对象是小模型不跑飞的核心）。
3. **每次 roll 重注人设 canon**（防 persona drift）：把人设卡 + 八字放 prompt，且在末尾再强调一次身份（recency anchoring）。
4. **worldbook 当硬护栏**：`selectEntries` 拿相关 canon；事件不许违反硬事实（如刀剑男士不会去便利店打工）。
5. **动态可发生类型集 + 硬禁止清单**（§4.9）：`event_type` 只能取当前 WorldContext 筛出的类型；硬禁止清单（重大伤亡、出本丸远行、引入新命名角色…）任何时候都不许碰。
6. **dry-run 模式**（仿 relay）：只 log 不写库，先调参观察 roll 质量。
7. （可选，后置）**canon 校验**：第二次便宜调用或 worldbook NOT-match 校验事件合规，不合规丢弃重 roll。MVP 先只 log 异常。

### 5.4 与真人聊天的衔接

已由 §4.6 解决：事件走 livingmemory 既有注入，和真人记忆同路径 → 自然不突兀。额外 `{life_state}` 让「此刻状态」一致。**关键不变量**：模拟器产出的记忆必须和真人对话产出的记忆**格式同构**（同样经 createMemory、同样字段），否则注入时风格割裂。

### 5.5 主动外联的雷区（MVP 含最小版，人设主导 + 机械底线）

> 研究最响的警报：HBS 审计 1200 次告别，**43% 用情感操控**（「你要走了？」「我离不开你」），用户骂「恶心/粘人」。Replika 因 love-bombing 被 FTC 投诉。**主动性优化参与度 = 必然读作粘人**。

故 MVP 的主动发消息 = **人设主导的判断 + 机械底线兜底**：

**决策交给人设（无 importance 门槛）**：roll 时那一次调用直接产出 `want_to_share.decision`（now/later/no）——**由角色性格决定想不想说、什么时候说**，而非硬编码数字阈值。髭切慵懒可能几天不主动，膝丸克制到几乎不开口，这本就该人设说了算。折进 roll 调用、零额外成本。system 里把六类操控话术（催促/挽留/情感勒索/FOMO…，见 §10 HBS）显式列为**禁止**，并要求 draft 只分享一件真发生的小事。

**三道与「重要度」无关的机械底线（纯熔断，防便宜模型抽风刷屏，即使 decision=now 也要过）**：
1. **安静时段**：在 `quietHours`（硬规则，默认 22:00–08:00）内一律不发，顺延到醒来。
2. **每日上限 + 最小间隔**：`proactiveDailyCap`（默认 1–2 条/天）、`proactiveMinIntervalHours`——纯熔断，撞到就压住，不是重要度判断。
3. **无回复抑制**：发出后 `proactiveSuppressAfterSilenceHours`（默认 3h）内无真人回复 → 抑制后续主动消息（ComPeer 模式，防连环夺命 call）。

**为什么保留这三道**：它们不是「重要度门槛」，是真人也会守的物理礼貌（不半夜发、不一天发二十条、没回就别追）。人设决定「想不想」，底线决定「能不能」。

- **世界模拟（常跑）与主动外联（极少）彻底分离**：绝大多数事件 `decision=no`，只静默进事件表/记忆。
- **发送通路**：经 relay 发给真人时天然吃到 relay 的限流/白名单/防环，无需重造。
- 默认 `proactiveEnabled` 可一键关；关掉就退化为纯被动浮现（最安全）。

## 6. bot 间自控聊天 + 高维主控（三期，单独熔断）

> 不进 MVP。成本爆炸与跑飞风险远高于单 bot；放最后，且独立开关。

### 6.0 关键架构决定：bot 互聊默认走「互聊代理」而非 relay 跑主模型窗口

**问题**：relay 驱动互聊时，每一轮都要各自 bot 的**主模型 + 完整人设预设窗口 + worldbook + 记忆注入**跑一遍——又贵又慢，让重人设管线为「背景闲聊」空转，还给主人设引入漂移风险。

**决定**：**relay 的定位是「bot 给真人/外部发真实 QQ 消息」**。而两个**自己的** bot 在后台闲聊、目的只是长出一条**共享记忆**——根本不需要真发 QQ 消息。故默认走**专用「互聊代理」（ChatProxy）**：内部模拟整段 A↔B 交流，产物是一条共享事件/记忆，不发真消息、不进真实窗口。

| | A) relay 驱动 | **B) 互聊代理（默认）** |
|---|---|---|
| 每轮成本 | 主模型 × 完整预设窗口 | **便宜模型 × 精简人设卡** |
| 在哪发生 | 真实窗口、真发 QQ 消息 | **内部模拟，不发真消息** |
| 产物 | 散在两窗口的消息 | **一条「我和膝丸聊了X」共享记忆** |
| 护栏 | 复用 relay 五层 | 轮数帽+TERMINATE+防 echoing（自带） |
| 适合 | 想让互聊**可见** | **纯背景生活模拟** |

- **B 完胜背景生活模拟目标**：便宜、快、可控、不污染真实窗口；产物正好是 scene-rules 同池共享的一条记忆 → 两个 bot 下次都能自然提起。
- **唯一取舍**：代理方案下互聊**过程不可见**，只落为记忆。若想偶尔「围观」，配 `visibleVia: relay` 切到 A 方案（此时才真发消息、真跑主模型）。
- **ChatProxy 产物**：一段对话摘要 → 经夜间归档路径 `createMemory` 写入**双方** presetId（或共享池），如同一次 roll 事件，只是 participants 是另一个 bot。

### 6.1 高维主控的职责边界

- **只做发起与节流，不做内容**：决定「此刻是否让 bot A 找 bot B」、给一个**话题种子**（来自共享事件/未了线/近期话题）、设**本次对话的硬上限**，然后把控制权交给 ChatProxy（默认）或 relay 链路（visible 模式）。
- **与 relay 分工**：relay = 对真人/外部发真实消息的管道与五层护栏；主控 = 决策层（何时、谁与谁、聊什么、聊几轮）。默认互聊不经 relay（走 ChatProxy 内部模拟）；仅 visible 模式才复用 relay。
- **素材来源**：模拟器的事件表 + life-state + 跨窗口互通记忆（scene-rules 已让多 bot 记忆可共享）。

### 6.2 防跑飞 / 防刷屏 / 防成本爆炸（研究背书）

- **硬轮数帽** `MaxMessageTermination`：一次 bot 互聊 ≤ K 轮（默认 4–6）。
- **语义完成信号**：约定 `TERMINATE`/自然收尾 token，任一方给出即停。
- **语义循环检测**：对近期消息哈希，重复即终止（治「车轱辘话」）。
- **防 echoing**（两 bot 互相同化，文献报告 15–25%）：ChatProxy 每轮在 prompt **末尾重注各自精简角色卡** + 结构化分隔「你的角色 / 对话历史」。
- **成本**：ChatProxy 用便宜模型 + 精简卡（比 relay 跑主模型省一个量级）；级联（低置信再升）；全局 `botChatDailyCap`。
- **防环**：ChatProxy 内部对话有明确 K 轮上限、天然收敛；visible 模式才需 relay 的 `myBots` 防环；主控额外记录「本轮对话 ID」避免 A→B→A 无限。

## 8. 分阶段 MVP

### Phase 1 — 单 bot 生活自循环（先做，最划算）
**先做这一环，因为它独立验证「不聊天世界也在长」，且 80% 价值在此。**
1. **持久化 Scheduler**（DB 表 + ready 扫描 + dispose 清理）——硬地基，先落。
2. **IdleWatcher**（订阅 message_collect）。
3. **WorldContext + EventRegistry**（世界上下文 + 动态事件类型注册表，§4.9）——EventRoller 的输入依赖。
4. **EventRoller**（便宜模型 + JSON schema + life-state 锚 + worldbook 护栏 + 动态类型集 + list-then-roll）；含 `fallbackToTemplate` 随机池兜底。
5. **LifeStateStore + EventLog**（两张表）。
6. **NightConsolidator**（主模型 → createMemory + runDream）。
7. **PromptProvider** 注入 `{life_state}`（可选 `{world_context}`）。
8. **最小 ProactiveBridge**（人设 want_to_share 主导 + 机械底线；可一键关）。
9. dry-run + debug + 本地模型默认。

**验收**：开一两天后，去看 `life_sim_event` 有连续合理、随天气/时段变化的事件流；去 livingmemory 看到夜间归档的新记忆；下次和 bot 聊天，它能自然提起这两天的事；主动消息符合人设节奏（髭切/膝丸各不同）且不粘人。

### Phase 2 — 打磨与扩面
- WeeklyPruner + importance 调参（多因子：novelty/thread-relevance）。
- canon 校验（不合规重 roll）。
- 多 preset/多 bot 并行；reflection 早触发（大事件即时整理，仿 Generative Agents importance>阈值）。
- 主动外联策略精化（opportune-moment、stochastic 阈值调参）。

### Phase 3 — bot 间自控互聊 + 高维主控
- Orchestrator（发起/节流/话题种子）。
- **ChatProxy 默认路径**：便宜模型 + 精简人设卡内部模拟 capped 互聊（轮数帽 + TERMINATE + 哈希防循环 + 末尾重注角色卡防 echoing），产物写成双方共享记忆——**不经 relay、不跑主模型窗口**（§6.0）。
- 可选 `visibleVia: relay` 让互聊真发消息、真跑主模型（想围观时）。
- 级联 + 全局 `botChatDailyCap`。
- 独立开关，默认关。

## 10. 参考（人机恋/agent 研究）

- **Generative Agents**（Park 2023）：memory stream + 检索三因子（recency·importance·relevance，等权 min-max 归一）+ reflection（importance 和 >150 触发，100 条→3 问→每问 5 洞见）+ top-down 规划。
- **AI Town**（a16z）：recency 衰减 0.99/h，importance 0–9，等权。
- **Letta sleep-time compute**：只读 primary + 后台 rewriter（`memory_rethink`），分层 cadence（per-session 去重 / session-end 轻整理 / weekly 重组 / monthly rollup）；「别过度归并、会丢粒度」。
- **Mem0**：ADD/UPDATE/DELETE/NOOP 冲突解决（新记忆与旧矛盾时）。
- **ComPeer**：事件检测 + 晨间反思 + importance 随机阈值门 + 3h 无回复抑制；对比固定规则被吐槽 6.6 条/天。
